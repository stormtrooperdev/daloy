# DaloyJS Roadmap

This document is the source of truth for what's shipped, what's next, and the order
we plan to ship it. It complements the [README](./README.md) (the pitch). Update
this file when scope changes — never let it drift behind reality.

**Versioning policy:** semver with a hard rule — every `0.x` minor bump may break
the public API; every `1.x` minor bump must not. Once `1.0.0` ships, deprecations
last at least one minor cycle before removal.

**Definition of done for every milestone:**

1. Implementation in `src/` with no `any` leaks across public types.
2. Tests added; `pnpm coverage` stays at **≥90% lines / 90% functions** with `pnpm coverage:branches` at **≥90% branches** (relaxed from 100% — see [AGENTS.md](AGENTS.md): on hard security work, ship the feature instead of chasing useless coverage of unreachable defensive branches or tsx phantom lines).
3. `pnpm typecheck`, `pnpm test`, `pnpm build`, and CI pass.
4. Security impact considered; `SECURITY.md` or threat notes updated when relevant.
5. Public-facing docs (`website/app/docs/...`) updated.
6. README "Status" table reflects new capability.

## Ongoing — `0.15.0`+ ("secure-by-default initiative")

Converts Daloy's existing security middleware from opt-in to opt-out where
exactly one correct default exists, and adds a small set of high-value
primitives. Full plan with risk matrix, breaking-change boundaries, and per-wave
test surface in [`otherdocs/secure-by-default-plan.md`](./otherdocs/secure-by-default-plan.md).
Wave 1 shipped in `0.15.0`, Wave 2 in `0.16.0`, Wave 3 in `0.17.0`, Wave 4
in `0.18.0`, Wave 5 in `0.19.0`, the Wave 4 leftover slice
(`loadShedding()`, `app.cspReportRoute()` + `secureHeaders({ reportingEndpoints, reportTo })`,
`disconnectStatusCode: 499` default, `defineConfig({ schema, source })`)
in `0.20.0`, and a focused Wave 5 leftover slice (`createJwtSigner()` /
`createJwtVerifier()` with `alg`-discipline + `exp`-required sign refusal,
`requireScopes()` with RFC-6750 challenge + per-request aggregation,
`etag()` helper with `Set-Cookie` / `Cache-Control: private | no-store | no-cache`
auto-skip) in `0.21.0`, and the Wave 5 auth-cohesive slice (`jwk()`
asymmetric-only JWKS middleware, `bearerAuth({ verify })`,
`basicAuth({ onAuthSuccess })`, and `Cache-Control: no-store` on auth 401
challenges) in `0.22.0`, and the Wave 5 remaining slice (`wsRateLimit()`,
`loginThrottle()`, `rotateSession()`, file-upload magic-byte guards,
`requirePayloadAuth`, and WebSocket safe defaults) in `0.23.0`; the remaining
waves below are still planned work.

- [x] **Wave 1 — additive (shipped in `0.15.0`):** the focused six-item slice scoped in [`otherdocs/secure-by-default-plan.md`](./otherdocs/secure-by-default-plan.md):

  1. **Default log redaction** in `createLogger()` for `authorization` / `cookie` / `set-cookie` / `x-api-key` / `password` / `token` / `secret` / OAuth-style token keys, plus any string value shaped like a JWT (`eyJ...`). Configurable via `{ redact: { keys, censor, useDefaults, redactJwtLikeStrings, maxDepth } | false }`; defaults exported as `DEFAULT_REDACT_KEYS`.
  2. **Strip `Server` and `X-Powered-By` from every response** — on by default in the response finalizer; opt out with `new App({ stripServerHeaders: false })`.
  3. **Reject duplicate `Host` / `Content-Length` headers** at the request boundary (HTTP-request-smuggling defense) via the new exported helper `assertNoDuplicateSingletonHeaders()` + `SMUGGLING_SINGLETON_HEADERS`; returns a structured `400 problem+json`.
  4. **`passwordHash()` / `passwordVerify()` helper** with no knobs at `@daloyjs/core/hashing` — Node `crypto.scrypt` with OWASP-aligned parameters (`N = 2^17`, `r = 8`, `p = 1`, 16-byte salt, 32-byte key) encoded as a self-describing PHC string. Verification rejects downgraded / malformed PHC parameters instead of silently accepting weaker hashes. Zero runtime dependencies.
  5. **Webhook HMAC verify helper** at `verifyWebhookSignature()` (+ `signWebhookPayload()` companion) — WebCrypto HMAC-SHA-256/384/512, constant-time comparison, accepts hex / base64 / base64url / matching GitHub-style `sha256=` / `sha384=` / `sha512=` prefixes / `Uint8Array`. SHA-1 and mismatched algorithm prefixes are deliberately rejected.
  6. **Explicit `app({ env: "production" | "development" | "test" })` option** that takes precedence over `NODE_ENV`; a one-time `warn` (`event: "env.mismatch"`) is logged through the pluggable logger when the explicit value disagrees with `NODE_ENV`. Threads through error redaction so `env: "production"` hides 5xx detail even without setting `NODE_ENV`.

  Non-breaking. The remaining expanded items listed below (constructor-poisoning rejection, `maxParamLength`/ReDoS guard, cooperative `AbortSignal`, request-id hardening, rate-limit `ban` escalation, CSRF user-bound tokens, bearer-auth strict RFC 6750, `[Symbol.asyncDispose]`, open-redirect guard, `TRACE`/`CONNECT` refusal, method-override refusal, ETag tracking guard, slow-loris adapter defaults, idle-timeout floors, generic session cookie name, first-class cookie helper with RFC 6265bis + CHIPS, `SameSite: 'Strict'` framework cookie default, `strictHeader` cookie parser, encrypted-and-signed cookie helper, body-reception timeout, cookie-helper password floor, `WWW-Authenticate` discipline, 401-vs-403 discipline, body-on-GET refusal, …) remain tracked here and will land in subsequent additive `0.15.x` patches or roll into the Wave 2 breaking-defaults release.
- [x] **Wave 2 — flip defaults (focused slice shipped in `0.16.0`, more landing in subsequent waves):** the focused four-item slice scoped in [`otherdocs/secure-by-default-plan.md`](./otherdocs/secure-by-default-plan.md) shipped first, breaking only the no-op cases that already had no `secureHeaders()` registered:

  1. **`secureHeaders()` auto-applied** on every `App` unless `secureDefaults: false` or `secureHeaders: false`. When a developer calls `app.use(secureHeaders(...))` later, the auto-installed instance is automatically dropped so the user's overrides win instead of being shadowed by the framework's defaults (detected via the new exported `SECURE_HEADERS_MARKER` symbol).
  2. **CORS deny-by-default for state-changing cross-origin requests** — `POST` / `PUT` / `PATCH` / `DELETE` requests carrying an `Origin` header from a different origin than the request URL are rejected with `403 problem+json` unless the matched route's `cors()` policy allows that origin (detected via the new exported `CORS_HOOK_MARKER` and `CORS_ORIGIN_ALLOW_MARKER` symbols). Read-only methods and same-origin / `Origin: null` / missing-`Origin` requests pass through. Opt out per feature with `app({ corsCrossOriginGuard: false })` or globally with `app({ secureDefaults: false })`.
  3. **Per-route Content-Type allowlist** — new `route({ accepts: ["application/x-www-form-urlencoded"] })` field overrides the global `allowedContentTypes` for a single route so legacy form-encoded handlers can opt in without loosening the default allowlist for the rest of the application.
  4. **Secure cookie defaults documented as already-met** — `session()` and `csrf()` already issue `Secure; HttpOnly; SameSite=Strict; __Host-` cookies; no behavioral change shipped for this item.

  The remaining expanded items (CORS `maxAge` policy, per-content-type JSON body cap default, CSP nonces default-on, multipart parts cap, Helmet header parity expansion, TLS enforcement, strict request-body validator, duplicate query keys, static-server defaults, immovable `frame-ancestors 'none'`, canonical trailing-slash policy, default `Cache-Control: no-cache`, payload-parsing `failAction: 'error'` reaffirmation, cookie-parsing `failAction: 'error'`, validator `failAction: 'error'` reaffirmation, response-schema validation default-on, response-shape filtering default-on, strict Content-Type checking default-on, conditional OpenAPI off in production, `redirectHttpToHttps()`, `rootPath` declarative model, …) remain tracked here and will land in subsequent additive `0.16.x` patches.
- [x] **Wave 3 — boot/first-request guards (focused slice shipped in `0.17.0`, more landing in subsequent waves):** the focused four-item slice scoped in [`otherdocs/secure-by-default-plan.md`](./otherdocs/secure-by-default-plan.md) shipped first:

  1. **Weak HMAC secret refuse-to-boot** — `session({ secret })` and any future helper that participates in the secret check (via the new exported `assertStrongSecret` / `MIN_PROD_SECRET_BYTES` / `WEAK_SECRET_STRINGS`) refuses to register in production when the secret is shorter than 32 UTF-8 bytes, matches a known-weak placeholder (`"changeme"`, `"your-jwt-secret"`, `"it-is-very-secret"`, …), or is a single-repeated-character string (`"a".repeat(64)`, `"0".repeat(64)`). Detection runs synchronously inside `app.use(...)` so the process exits during startup, not on first request. Detected via the new exported `SESSION_HOOK_MARKER` + `SESSION_SECRETS_MARKER` symbols on the `Hooks` object so third-party session implementations can opt in.
  2. **`cors({ origin: "*" })` refuse-to-boot in production** — synchronous check inside `app.use(...)`; refuses both `origin: "*"` and `origin: ["a.example", "*"]`. Detected via the new exported `CORS_WILDCARD_ORIGIN_MARKER` symbol stamped on the CORS hook layer.
  3. **`session()` + state-changing route without `csrf()` refuse-to-boot** — first-request boot guard: when any route accepts `POST` / `PUT` / `PATCH` / `DELETE` AND a `session()` hook is installed AND no `csrf()` hook is installed, the framework throws on first dispatch and caches the boot error so every subsequent request rethrows the same failure. Opt out per app with `app({ csrf: "off" })` (acknowledges that the app is not browser-facing) or globally with `app({ secureDefaults: false })`. Detected via the new exported `CSRF_HOOK_MARKER` symbol.
  4. **First request with `X-Forwarded-*` and `trustProxy` unset returns 500** — per-request guard: when the app was constructed without an explicit `app({ trustProxy: true | false })` value and any request carries `X-Forwarded-For`, `X-Forwarded-Host`, `X-Forwarded-Proto`, `X-Forwarded-Port`, or `X-Real-IP`, the framework refuses to dispatch (rate-limit / audit-log / request-id propagation would otherwise honour the attacker-supplied IP). Logged at `warn` exactly once per process via a latch; the response is a structured `500 problem+json`. Opt out with `app({ trustProxy: true })` (running behind a trusted proxy), `app({ trustProxy: false })` (explicitly ignore forwarded headers), or `app({ secureDefaults: false })`.

  All four guards are gated on the resolved environment being `production` (sources: `app({ env: "production" })`, then `app({ production: true })`, then `NODE_ENV === "production"`) so CI / staging surfaces that ship sample secrets while iterating do not pay the refuse-to-boot cost. The remaining expanded items (`requestTimeout: 0` refuse-to-boot, `allowedHosts` policy, `defaultAuth` server-wide guard, `globalDependencies` typed-DI conflict check, `canonicalHostRedirect`, `strictRequestBodies`, env-var-only `trustedProxies: '*'` refusal, …) remain tracked here and will land in subsequent additive `0.17.x` patches.
- [x] **Wave 4 — lifecycle & health (focused slice shipped in `0.18.0`, more landing in subsequent waves):** the focused four-item slice scoped in [`otherdocs/secure-by-default-plan.md`](./otherdocs/secure-by-default-plan.md) shipped first:

  1. **Connection-draining graceful shutdown** — `app.shutdown(timeoutMs, reason?)` (alias `app.close()`) flips a `draining` flag synchronously so every subsequent request gets a structured `503 problem+json` carrying `retry-after: 5` and `connection: close`; in-flight responses that finish while draining also gain `connection: close` so HTTP/1.1 load balancers stop reusing the keep-alive socket. The Node adapter registers an idle-connection close hook (`server.closeIdleConnections()`) that fires the moment draining begins, killing keep-alives without an in-flight request without affecting requests still being handled. Exposed via the new `_registerIdleConnectionCloseHook` internal hook for custom adapters.
  2. **Crash-on-unhandled-rejection in production** — new `app({ crashOnUnhandledRejection })` option: when omitted, defaults to `true` in production (`env: "production"` or `NODE_ENV === "production"`) and `false` elsewhere; explicit `true` installs the listeners outside production, explicit `false` skips them entirely. Both `unhandledRejection` and `uncaughtException` log `fatal` through the pluggable logger and call `process.exit(1)`, deliberately avoiding the "swallow and keep running" anti-pattern. Process-wide latch (`_resetCrashHandlersForTests` exposed for tests) ensures the listeners are installed exactly once even when multiple `App` instances boot in the same process. No-op on runtimes without `process.on` (Workers / Edge / Fastly).
  3. **`app.healthcheck()` / `app.readinesscheck()` primitives** — opt-in route registration with sensible defaults: paths `/healthz` and `/readyz`, per-IP fixed-window rate limit (60 requests / 60 s, in-memory, pass `rateLimit: false` to disable), optional `Authorization: Bearer <token>` requirement compared via `timingSafeEqual`. In production under `secureDefaults`, registration without `token` refuses-at-construction with a structured error unless `acknowledgeUnauthenticated: true` is also declared (info-disclosure surface). Readiness returns `503` with `retry-after: 5` while draining OR while any plugin is still pending.
  4. **Signal-driven shutdown on the Node adapter** — existing `serve({ handleSignals: true })` behavior reaffirmed: handlers are installed via `process.once(...)`, and `app.close()` is idempotent so any caller that races a signal-driven shutdown sees the same drain semantics.

  The remaining expanded items (`loadShedding()`, CSP report-route + `Reporting-Endpoints` wiring, `disconnectStatusCode: 499` default, `defineConfig({ schema, source })` boot-time typed configuration validation, …) remain tracked here and will land in subsequent additive `0.18.x` patches.
- [x] **Wave 4 — leftover expanded items (shipped in `0.20.0`):** the four-item slice landed:

  1. **`loadShedding()` primitive** — first-party `under-pressure`-equivalent middleware exported from `@daloyjs/core`. Returns `Hooks` so `app.use(loadShedding({ maxEventLoopDelayMs, maxEventLoopUtilization, maxHeapUsedBytes, maxRssBytes, retryAfterSeconds, healthCheck }))` short-circuits every incoming request with `503 problem+json` + `Retry-After` once any configured threshold trips. extending custom HTTP methods (e.g. WebDAV `MKCOL`, `COPY`, `PROPFIND`, `MOVE`) requires explicit opt-in via `app({ extendedMethods: ['MKCOL', 'COPY'] })` AND refuses-at-construction in production under `secureDefaults` unless `acknowledgeExtendedHTTPMethods: true` is also declared (custom methods bypass the Wave 1 `TRACE` / `CONNECT` refusal, the Wave 2 strict-Content-Type check on state-changing methods, the Wave 9 audit item-26 body-on-`GET` / `HEAD` refusal — every custom method must be explicitly classified by the developer as state-changing or read-only, with a structured error on misclassification). **`rewriteUrl` single-hook refusal-on-second-registration + pre-security-check ordering** — the URL-rewrite hook (companion to the Wave 9 audit item 9 immutable-URL audit) runs exactly once per request AND ALWAYS BEFORE every security middleware (auth, CSRF, secure-headers, rate-limit, request-id propagation); refuse-at-registration when a second `rewriteUrl` hook is declared (only one allowed per app), and refuse-at-registration when any security middleware declares an ordering dependency that would place it BEFORE the rewrite hook (the only correct ordering is: rewrite → security checks → handler — the upstream framework documents this as developer-burden ordering but ships it without refusing the misconfiguration). **`addContentTypeParser` custom-parser proto-poisoning guard** — registering a custom content-type parser for `application/json`, `text/json`, any `+json` vendor suffix, or any registered JSON-shaped media type refuses-at-construction in production under `secureDefaults` when the custom parser does NOT explicitly declare it preserves the Wave 1 proto-poisoning / constructor-poisoning guard (the upstream framework documents the `getDefaultJsonParser('ignore', 'ignore')` opt-out as a feature — DaloyJS refuses the opt-out in production because there is no scenario where bypassing the guard is correct on a production JSON parser). **Per-request `AsyncLocalStorage` `ctx.requestContext` primitive** — first-class typed per-request context store wired into every adapter (Node / Bun / Deno / Cloudflare Workers / Vercel Edge / AWS Lambda — every adapter that exposes `AsyncLocalStorage` or an equivalent) so developers stop reaching for module-level state to thread the current user / tenant / trace ID through helpers and library code (a documented cross-request data-leak class where developer code mutates a module-level variable in handler A and reads the wrong value in concurrent handler B); typed via `AppState` so `ctx.requestContext.user` is the same type as `ctx.state.user`; refuses-at-construction `ctx.requestContext` access from a non-request scope (handler-outer code, plugin-level state) in production under `secureDefaults` to prevent the inverse "developer read `ctx.requestContext` at module top-level and got `undefined` shared across every request" footgun. **Raw-body access for HMAC verify + memory-amplifier refusal** — the body parser exposes `ctx.rawBody` (a `Uint8Array` snapshot of the pre-parse bytes) only on routes that declare `rawBody: true` AND declare a webhook-HMAC-verify scheme (or a content type that requires byte-exact verification); refuse-at-construction `rawBody: true` without a declared HMAC verify route in production under `secureDefaults` (the upstream framework ships raw-body access as a flag with no companion declaration, so developers enable it for one route and accidentally amplify every other route's memory footprint by 2× — DaloyJS refuses the unused enablement). **Reverse-proxy direct-exposure refusal** — `app({ behindProxy: 'none' })` AND listening on `0.0.0.0` in `env: 'production'` refuses-to-boot with a structured error naming the documented anti-pattern ("Node.js HTTP servers should be deployed behind a reverse proxy for TLS termination, slow-loris defense, header sanitization, and horizontal scalability"); explicit opt-out via `acknowledgeDirectInternetExposure: true` is permitted but emits a once-per-process `error` log naming every default the framework cannot enforce when directly exposed to the public internet (the upstream framework documents this as recommendation prose — DaloyJS refuses the recommendation's opposite by default). **`return503OnClosing` always sets `Connection: close`** — the existing Wave 4 graceful-shutdown default already returns `503` to requests arriving during shutdown; this Wave 12 reaffirm tightens the spec: the `503` response ALWAYS carries `Connection: close` (HTTP/1.1) AND `GOAWAY` (HTTP/2) so the load balancer immediately stops routing new connections to the closing instance, regardless of the `keep-alive` policy the developer configured for normal traffic (closes the "load balancer kept the dying instance in rotation because the framework forgot the `Connection: close` on the shutdown-503" surface). **Wave 9 audit-checklist additions xxxviii–xlvi** — the standing audit absorbs the new items above so the bake-ins cannot regress: plugin-boot-deadline refusal audit, semicolon-delimiter refusal audit, error-handler-override refusal audit, structured-problem-json clientErrorHandler / onBadUrl audit, no-client-request-id-by-default audit, 404-runs-through-security-chain audit, validator `allErrors: false` production-default audit, extended-HTTP-methods allowlist audit, `rewriteUrl` single-hook + pre-security ordering audit. The intent of Wave 12 is to close the boot-time, lifecycle, parser, and router surfaces the original first-pass review of this framework did not enumerate — each item is a one-line developer-burden config in the upstream framework that DaloyJS removes-the-knob-on by picking the safe default and refusing the foot-shooting opt-out in production.

## Stabilization — `1.0.0` ("public API freeze")

Ship date target: when the items below are simultaneously true. We'd rather
delay `1.0.0` than freeze the wrong API.

- [ ] No breaking change in two consecutive `0.x` minors.
- [ ] At least three production users on file (internal + external).
- [ ] Public benchmark suite published with reproducible numbers.
- [ ] Migration guide from the most-used Node frameworks.
- [ ] Security policy and disclosure process have been exercised at least once.
- [ ] Branch coverage has a stable high-confidence gate; any ignored branches have documented runtime or source-map reasons. — Gate is shipped (`pnpm coverage:branches`); ratchet to `>= 98%` is the open subtask under `0.10.0`.

## Out of scope (intentional)

Avoiding scope creep is part of the design. These are explicit non-goals:

- A built-in ORM or query builder.
- A bundled UI / view layer.
- Project-wide DI containers.
- Anything that requires patching `globalThis` or monkey-patching `Request`/`Response`.

## How to propose a roadmap change

1. Open an issue titled `roadmap: <change>`.
2. Describe the user-visible problem first; design comes second.
3. Reference the milestone you'd like the work to land in and why.
4. If accepted, open a PR that updates this file in the same change as the implementation.

The maintainers explicitly reserve the right to defer or drop items based on
real usage signal — the roadmap is a plan, not a contract.
