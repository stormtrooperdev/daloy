# DaloyJS Roadmap

This document is the source of truth for what's shipped, what's next, and the order
we plan to ship it. It complements the [README](./README.md) (the pitch). Update
this file when scope changes — never let it drift behind reality.

**Versioning policy:** semver with a hard rule — every `0.x` minor bump may break
the public API; every `1.x` minor bump must not. Once `1.0.0` ships, deprecations
last at least one minor cycle before removal.

**Definition of done for every milestone:**

1. Implementation in `src/` with no `any` leaks across public types.
2. Tests added; `pnpm coverage` stays at **100% lines / 100% functions**.
3. `pnpm typecheck`, `pnpm test`, `pnpm build`, and CI pass.
4. Security impact considered; `SECURITY.md` or threat notes updated when relevant.
5. Public-facing docs (`website/app/docs/...`) updated.
6. README "Status" table reflects new capability.

## Current — `0.2.x` ("confidence & lifecycle")

Goal: make every change safer to ship before expanding the feature surface.
None of these break the existing public API.

- [x] **`onSend` hook** symmetric to `beforeHandle` for response transformation.
- [x] **GitHub Actions CI** running install, typecheck, tests, coverage, build, and audit.
- [x] **Security policy** (`SECURITY.md`) and vulnerability disclosure process.
- [x] **Release pipeline hardening**: protected npm publish environment, OIDC trusted publishing with provenance, blocked egress on publish jobs, and static workflow/security scanners.
- [x] **Project scaffolder** (`pnpm create daloy`) shipped as `packages/create-daloy` with `node-basic`, `vercel-edge`, and `cloudflare-worker` templates.
- [x] **Docs discoverability + integration docs**: per-page metadata, sitemap, robots, OpenGraph image, and ORM guides in `website`.
- [x] **Branch coverage gate** — dist-based `pnpm coverage:branches` (compiled JS, no tsx source-map noise) enforced in CI at `>= 95%`; established as the stable, ratchetable floor. See `tsconfig.coverage.json` and the `coverage:branches` script.
- [x] **Docs cleanup**: publish a maintainer-facing release checklist (in `SECURITY.md`) and keep package naming/examples aligned with `@daloyjs/core`.

**Exit criteria:** every item above either ships or is moved to a later milestone
with an explicit reason. No silent dropouts.

## After — `0.3.x` ("streaming & observability")

Streaming and tracing are adapter-sensitive, so each item starts with an API
design issue before implementation.

- [x] **Streaming response helpers**: SSE + NDJSON with backpressure-safe writers (`sseStream`, `sseResponse`, `ndjsonStream`, `ndjsonResponse`).
- [x] **OpenTelemetry tracing hook** (`otelTracing`): per-request `SERVER` span with HTTP semantic-convention attributes; ends in `onSend`, escalates 5xx and thrown handlers to `ERROR`.
- [x] **OpenAPI extras**: `securitySchemes` builders (`httpBearerScheme`, `httpBasicScheme`, `apiKeyScheme`, `oauth2Scheme`, `openIdConnectScheme`), top-level `webhooks`, per-operation `callbacks`, and `discriminator` / `discriminatedUnion` helpers.

## Then — `0.11.0` ("real-time")

- [x] **WebSocket primitives** with adapter coverage for Node and Bun — runtime-agnostic RFC 6455 frame protocol in `src/websocket.ts`, typed `app.ws(path, handler)` registration, `defineWebSocket()` helper, and adapter wiring for `@daloyjs/core/node` and `@daloyjs/core/bun`. ## Soon — `0.14.0`+ ("secure-by-default initiative")

exactly one correct default exists, and adds a small set of high-value
primitives. Full plan with risk matrix, breaking-change boundaries, and per-wave
test surface in [`otherdocs/secure-by-default-plan.md`](./otherdocs/secure-by-default-plan.md).
Awaiting owner sign-off before implementation begins.

JWT/JWK/OAuth/OIDC strategies, RBAC/ABAC models, AES recipe, DI guards/interceptors model.

- [ ] **Wave 10 — zero-runtime-dependency batteries-included parity & governance audit (target `0.20.x`):** new wave that converts the deep-dive against a zero-runtime-dependency, batteries-included framework (recorded in [`otherdocs/security_task.txt`](./otherdocs/security_task.txt)) into a standing posture covering the cross-cutting items that do not slot cleanly into a single primitive of waves 1–9. Items are policy + audit gates, not one-time changes. **Recurring security-disclosure exercise** — the `1.0.0` exit criterion "Security policy and disclosure process have been exercised at least once" is converted into a recurring gate; the disclosure rotation is documented in a `SECURITY-CONTACTS` file (humans on rotation, pager handoff cadence) and the rotation is tested with a simulated disclosure exercise at least once per quarter, recorded in `PROJECT_HISTORY.md`. **Zero-runtime-dependency posture audit** (CI grep gate) — the `dependencies` block of `@daloyjs/core/package.json` is empty in every release; adapter-specific deps live behind subpath exports as peer-deps, validator deps live as peer-deps; companion to the Wave 8 governance stance and the Wave 9 audit item 19. **Refuse-to-publish on unverified contributor** — the release workflow refuses to publish when the actor is not on the documented `SECURITY-CONTACTS` rotation AND not on the GitHub-organization 2FA-enforced list; complements the existing protected `npm-publish` environment, the OIDC trusted-publishing path, and the Wave 8 mandatory-2FA item. **Transitive-dep audit** — even when a new runtime dep is approved through the Wave 8 governance gate, its own runtime dep graph is audited at PR review time; the framework refuses to take on a runtime dep whose transitive runtime closure is larger than zero unless `SECURITY.md` records an explicit waiver naming every transitive dep. **Plugin-prerequisite enforcement audit** — companion to the Wave 9 audit items 16 and 18; CI gate asserts the framework refuses-to-boot at the end of plugin registration when any declared `dependencies: string[]` entry is missing, AND asserts no first-party plugin reads cross-plugin state without a declared dependency. **Extension-order determinism audit** — companion to the Wave 6 `before` / `after` ordering; CI gate asserts the order in which security middleware (auth, CSRF, secure-headers, rate-limit) executes is identical regardless of plugin-registration order, and that adding a new plugin can never reorder the existing security stack unless the new plugin declares an explicit `before` / `after` relationship to it. **Governance floor reaffirmed** — every item already shipped in `.github/` (SHA-pinned actions, OIDC trusted publishing with provenance, harden-runner, blocked egress on publish jobs, CODEOWNERS on privileged files) is now the documented governance floor: removal of any one of these requires a `SECURITY.md` entry justifying the removal and a maintainer-quorum sign-off in the PR. The intent of Wave 10 is to make every Wave 1–9 default permanent: a future contributor cannot quietly add a runtime dep, quietly skip plugin-dependency declarations, or quietly reorder the security stack, because each of those actions trips a CI gate that names the change and routes it through review.

### Feature-comparison provenance

The wave items above are derived from a head-to-head review of established JS
frameworks and their security plugin ecosystems, recorded in
[`otherdocs/security_task.txt`](./otherdocs/security_task.txt). The latest
four additions to that review cover (a) a pattern-agnostic, TS / Bun-first
ergonomic framework whose documented defaults (wildcard CORS on, credentials
on, plaintext cookies, `JWT alg: HS256` + JWK without an asymmetric-only
path, signed-cookie rotation that accepts unsigned fallback, `Server-Timing`
default-on, `128 MB` blanket body cap, opt-in production validation-detail
leak, Bearer plugin that explicitly declines verification), (b) a
minimalist async-middleware framework whose stated value is "small and
unopinionated" but whose unstated cost is that every single security
primitive — body cap, CSRF, secure headers, rate limit, cookies,
request-id, log redaction, graceful shutdown, timeouts, IP spoofing
defense, open-redirect guard, header-injection guard — is the developer's
problem to find, vet, wire up in the right order, and keep current, (c)
a zero-runtime-dependency, batteries-included framework whose documented
defaults (cookie `SameSite: 'Strict'`, cookie `strictHeader: true` parsing,
first-class encrypted-and-signed cookies, `Cache-Control: no-cache` on every
response, payload / validator / cookie-parsing `failAction: 'error'`,
response-schema validation default-on with `sample: 100` in dev, per-request
auth `verify()` revalidation hook, scheme-level `requirePayloadAuth: true`
that routes cannot disable, server-wide default auth strategy with
explicit per-route opt-out, `disconnectStatusCode: 499` for client-aborted
requests, lazy `info.remote` to prevent IP enumeration in logs, `internal:
true` routes unreachable from the HTTP listener, plugin `dependency([...])`
refuse-to-boot, namespace-protected decorators, explicit `before` / `after`
extension ordering, zero external runtime dependencies as a governance
stance, mandatory 2FA for every contributor with publish access), and (d) a
contract-first, dependency-injection-driven, type-hint-validated framework
whose documented defaults (strict `Content-Type` checking default-on so JSON
bodies without the header are refused at the framework boundary,
response-shape filtering against the declared response schema as the
documented security primitive, declarative typed `Security(scheme,
scopes=[...])` whose OAuth2 scope requirements are aggregated across the
dependency tree and automatically surfaced in the per-operation OpenAPI
`security: [...]` block, RFC-6750-aligned `WWW-Authenticate` challenges
including `error="invalid_token"` / `error_description="..."` on failures,
semantically correct `401` vs `403` status codes with a dedicated docs page
documenting the legacy `403`-for-missing-auth pattern as an anti-pattern,
opt-in `--forwarded-allow-ips` for trusting `X-Forwarded-*` headers, `root_path`
declarative model for reverse-proxy-stripped path prefixes, conditional
OpenAPI via env var with the explicit "security through obscurity" warning,
server-wide global typed dependencies via `dependencies=[Depends(...)]`,
`HTTPSRedirectMiddleware` / `TrustedHostMiddleware` / `GZipMiddleware`
one-line helpers, dummy-hash compare on missing user to defeat login
enumeration via timing, `secrets.compare_digest` for every secret
comparison) are the most thorough "every security check is a typed,
composable, declarative dependency" stance reviewed in this file and feed
every Wave 1 / 2 / 3 / 4 / 5 / 6 / 7 / 8 addition flagged above as well as
the Wave 9 audit-list extensions (items 20–25). The rule of inclusion is
unchanged: a feature only becomes a default when exactly one correct value
exists and DaloyJS can pick it without removing developer choice on policy
questions.

## Stabilization — `1.0.0` ("public API freeze")

Ship date target: when the items below are simultaneously true. We'd rather
delay `1.0.0` than freeze the wrong API.

- [ ] No breaking change in two consecutive `0.x` minors.
- [ ] At least three production users on file (internal + external).
- [ ] Public benchmark suite published with reproducible numbers.
- [ ] Migration guide from the most-used Node frameworks.
- [ ] Security policy and disclosure process have been exercised at least once.
- [ ] Branch coverage has a stable high-confidence gate; any ignored branches have documented runtime or source-map reasons. — Gate is shipped (`pnpm coverage:branches`); ratchet to `>= 98%` is the open subtask under `0.10.0`.

## Out of scope (intentional)

Avoiding scope creep is part of the design. These are explicit non-goals:

- A built-in ORM or query builder.
- A bundled UI / view layer.
- Project-wide DI containers.
- Anything that requires patching `globalThis` or monkey-patching `Request`/`Response`.

## How to propose a roadmap change

1. Open an issue titled `roadmap: <change>`.
2. Describe the user-visible problem first; design comes second.
3. Reference the milestone you'd like the work to land in and why.
4. If accepted, open a PR that updates this file in the same change as the implementation.

The maintainers explicitly reserve the right to defer or drop items based on
real usage signal — the roadmap is a plan, not a contract.
